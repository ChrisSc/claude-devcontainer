# observability — audit findings

The container boots through a load-bearing ordered pipeline (firewall → seed → `claude update` → cron), and every phase mutates real, security-relevant state — yet none of it produces a persistent, structured, timestamped event trail. All boot scripts log free-text to stdout only, which lives solely in ephemeral `docker compose logs` with no boot ID to disambiguate the entrypoint run from the devcontainer re-run. There is no event-completeness test asserting the lifecycle ran end-to-end, no timestamps on boot lines, no structured-event convention for scheduled cron agents, and the firewall derives rich apply-state (CIDR counts, resolution failures, verify results) only to discard it. None of these are crashing bugs — the container boots and runs — but collectively the area falls short of the org observability-first standard: no JSONL journal, no typed taxonomy, no completeness gate. All five findings are medium-or-lower severity and share a single remediation spine: introduce a fire-and-forget JSONL event helper and build the other checks on top of it.

## Boot scripts log free-text to stdout only; no persistent, structured JSONL boot-events trail

- **Severity / kind:** medium / maintainability
- **Location:** [.devcontainer/entrypoint.sh:12](.devcontainer/entrypoint.sh#L12)
- **Evidence:**
  > entrypoint.sh:12 `echo "[entrypoint] applying firewall (FIREWALL_MODE=${FIREWALL_MODE:-strict})"`; init-firewall.sh:65 `log()  { echo "[firewall] $*"; }`; init-cron.sh:25 `log()  { echo "[cron] $*"; }`; seed-claude.sh:20 `echo "[seed] installed CLAUDE.md"`. `rg -i 'jsonl|event' --type-not md` returns nothing — no structured event stream exists anywhere.
- **Why it matters:** The org observability standard mandates a structured, append-only, ISO-timestamped + sequenced JSONL event journal for every state-changing operation, persisted so it survives the process. Every boot phase here mutates real state (iptables policy/ipset membership, ~/.ssh symlink, crontab install, cron daemon start) yet emits only prefixed free-text to stdout. That output lives solely in `docker compose logs` (Makefile:20-21 `logs: $(COMPOSE) logs -f`), which is ephemeral: it is lost on `docker rm`/`make down`/`make nuke`, rotated by the daemon, and interleaved across the entrypoint run AND the devcontainer.json:56 `postStartCommand` re-run with no way to tell the two boots apart. There is no greppable key=value form and no boot ID, so 'why did egress break at 03:00 last night' is unanswerable after a restart. ENVIRONMENT.md (seed-claude.sh:46-86) is a human snapshot that is overwritten each boot, not an append-only trail.
- **Recommendation:** Add a tiny shared `log-event.sh` helper sourced by entrypoint.sh, init-firewall.sh, seed-claude.sh, and init-cron.sh that appends one JSON object per line to a persistent path on the claude-config volume, e.g. `$CLAUDE_CONFIG_DIR/logs/boot-events.jsonl`. Emit `{ts, seq, boot_id, phase, event, ...payload}` where `ts` is `date -u +%FT%TZ`, `boot_id` is generated once in entrypoint.sh and exported (also threaded into the postStartCommand re-run so VS Code boots correlate), and `seq` is a monotonic counter. Keep the existing `echo`/`log()` console lines as the dev mirror; the JSONL append is fire-and-forget (`>> file || true`) so it never blocks or fails the boot. Namespace events dot-style per the standard: `firewall.apply.start`, `firewall.degraded`, `seed.ssh.linked`, `cron.daemon.started`, etc.

## No event-completeness / lifecycle smoke test asserting the boot sequence ran end-to-end

- **Severity / kind:** medium / test-gap
- **Location:** [Makefile:6](Makefile#L6)
- **Evidence:**
  > `fd -i 'test|spec'` finds zero test files or directories in the repo. Makefile `.PHONY` (line 6) lists `up shell rebuild logs stop down nuke firewall doctor cp-skill cron-reload cron-log env allowlist db-*` — there is no `test`, `smoke`, or `boot-check` target. The only post-boot assertion is inside the firewall itself (init-firewall.sh:316 `curl ... https://example.com` negative test), and nothing audits that ALL four boot phases completed.
- **Why it matters:** The org standard requires event-completeness tests that 'audit a full lifecycle ... verify correct event ordering, no sequence gaps, required events present.' The boot here is a load-bearing ordered pipeline (entrypoint.sh:4-5 'Ordering is load-bearing'): firewall -> seed -> claude update -> cron. A regression that silently skips a phase (e.g. seed-claude.sh exits 0 early, or cron never starts) is invisible — non-fatal `|| echo WARN` wrappers (entrypoint.sh:30,38) mean a broken phase doesn't fail the boot, so CI/the user only finds out when SSH keys vanish or cron jobs never fire. There is no automated check that the lifecycle is intact.
- **Recommendation:** Once the JSONL trail from the prior finding exists, add a `make boot-check` (or `make smoke`) target that execs into claude-code, tails `~/.claude/logs/boot-events.jsonl` for the latest `boot_id`, and asserts the required events are present and ordered for the chosen FIREWALL_MODE: `firewall.apply.start` -> (`firewall.complete.strict` | `firewall.complete.permissive` | `firewall.degraded`), `seed.environment.regenerated`, `seed.ssh.linked`, `cron.installed`, `cron.daemon.started`, and `entrypoint.ready`. Fail with the first missing/out-of-order event. This is the shell-adapted form of an event-completeness test and gives the repo its first lifecycle gate.

## Boot log lines carry no timestamp, so durations and time-of-failure are unrecoverable from captured output

- **Severity / kind:** low / maintainability
- **Location:** [.devcontainer/entrypoint.sh:41](.devcontainer/entrypoint.sh#L41)
- **Evidence:**
  > `rg -n 'date |timestamp|%Y|EPOCHSECONDS' .devcontainer/*.sh` finds no timestamp emission in any boot script. entrypoint.sh:41 final line `echo "[entrypoint] ready — claude ..."` has no time; init-firewall.sh's retry/fetch logs (`warn "GitHub meta fetch attempt ${attempt} failed; retrying"` :211) carry no elapsed time. `docker compose logs` can prepend timestamps only when invoked with `-t`, which Makefile:21 (`$(COMPOSE) logs -f`) does not pass.
- **Why it matters:** The standard requires every event to carry an ISO timestamp. Without one, you cannot tell how long the firewall fetch retries took, when in wall-clock time `claude update` stalled, or correlate a boot-time failure against an external incident (e.g. a GitHub meta outage). The firewall's 3x retry loops and the network-dependent `claude update` are precisely the slow/flaky steps where timing telemetry matters, and that signal is simply absent today.
- **Recommendation:** When adding the JSONL helper (finding 1), include a `date -u +%FT%T.%3NZ` field on every event — that satisfies the requirement cleanly. As a low-effort interim for the human console mirror, prefix the `log()`/`warn()` helpers (init-firewall.sh:65-66, init-cron.sh:25-26) and the entrypoint `echo`s with `$(date -u +%H:%M:%S)`, and add `-t` to the `make logs` target so docker-captured output is timestamped. Note this is a per-line clock, not a duration; the JSONL `ts` + `seq` pair is what enables real elapsed-time analysis.

## Cron job logging guidance is ad-hoc per-job redirection with no structured-event convention for scheduled agents

- **Severity / kind:** low / docs
- **Location:** [.devcontainer/seed/crontab:13](.devcontainer/seed/crontab#L13)
- **Evidence:**
  > seed/crontab:13-14 'Logs: cron's own daemon output is not captured (no syslog) — redirect each job's output to ~/.claude/cron/logs/'; example job (:27) `... >> /home/claude/.claude/cron/logs/daily.log 2>&1`. init-cron.sh creates `LOG_DIR="$CRON_DIR/logs"` (:21,35) but writes nothing structured into it; `make cron-log` (Makefile:41-42) just `tail -F ~/.claude/cron/logs/*`. There is no record of a job's start/exit-code/duration — only whatever free-text the agent itself prints.
- **Why it matters:** Scheduled Claude agents are state-changing operations the standard says must emit structured events, yet the only observability is 'append stdout to a flat .log per job.' There is no machine-readable record of whether a 09:00 job actually fired, what exit code it returned, or how long it ran — so a silently failing nightly agent (e.g. egress blocked, auth expired) leaves no greppable trail and no completeness signal. Daemon-level cron output is explicitly uncaptured (seed/crontab:13), widening the blind spot.
- **Recommendation:** Ship a tiny `cron-run` wrapper (installed beside crontab-edit/crontab-reload in Dockerfile COPY :144-145) that jobs invoke as `cron-run <job-name> -- claude -p '...'`; it appends one JSONL line to `~/.claude/cron/logs/jobs.jsonl` on start and on exit (`{ts, job, event:'cron.job.start'|'cron.job.end', exit_code, duration_s}`) while still tee-ing the job's own stdout to the existing per-job log. Update the seed/crontab example and the README/seed 'Scheduled agents' sections to recommend it. This gives scheduled agents the same typed-event treatment as the boot path and makes `make cron-log` show a real run history, not just raw output.

## Firewall computes rich apply-state (CIDR/IP counts, resolution failures, verify result) but discards it instead of recording a queryable summary

- **Severity / kind:** low / maintainability
- **Location:** [.devcontainer/init-firewall.sh:320](.devcontainer/init-firewall.sh#L320)
- **Evidence:**
  > init-firewall.sh already counts everything: `log "added ${added} IP(s) for ${domain}"` (:117), `log "added ${count} AWS CIDR(s)"` (:159), `log "added GitHub meta ranges"` (:219), `warn "could not resolve ${domain} — skipping"` (:108), and the verify block `log "verified: example.com is blocked"` (:320) / `log "verified: api.github.com is reachable"` (:325). All of it goes to stdout only. The effective mode IS persisted machine-readably (`printf '%s\n' "$FIREWALL_MODE" > "$MODE_FILE"` :101) — establishing the pattern — but the apply *outcome* is not.
- **Why it matters:** The firewall is the single most security-relevant state mutation in the container, and its result is exactly what an operator or an in-container agent needs to introspect: how many CIDRs loaded, which domains failed to resolve, whether the GitHub/AWS feeds succeeded, and whether the negative+positive verify passed. Today `make firewall` (Makefile:32-33) runs the script but surfaces none of this as a result an agent can read — `seed/CLAUDE.md` even tells agents to debug egress via `sudo ipset list allowed-domains` (a raw dump) because no summarized state exists. The mode-file precedent (init-firewall.sh:24-28,99-101) proves a small `/etc/claude-firewall/` state file is already the idiomatic mechanism here.
- **Recommendation:** Have init-firewall.sh write a `/etc/claude-firewall/status.json` at the end of each run (alongside the existing `mode` file write at :101) capturing `{ts, mode, github_ranges_loaded, aws_cidrs, ipset_size: $(ipset list allowed-domains | grep -c '^[0-9]'), unresolved_domains:[...], verify_blocked_ok, verify_github_ok}`. Add `make firewall-status` to `cat` it from the host. This turns the data the script already derives into a queryable artifact for agents and operators, costs ~10 lines, and stays consistent with the existing mode-file pattern (keep resolver/verify behavior non-fatal per CLAUDE.md).
