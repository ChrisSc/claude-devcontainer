# db — audit findings

The db area is functional but its lifecycle and backup tooling is unreliable around the opt-in, profiled Postgres sidecar. The most serious problem is that the destructive `make nuke` target silently fails to destroy DB data (and leaves a dangling network) whenever the sidecar is running, directly contradicting its own docstring — a "destroy that doesn't destroy". A cluster of related Makefile gaps follow from the same root cause: lifecycle and helper targets omit `--profile db`, so `stop`/`down` orphan `claude-db`, and the `db-*` helpers assume `claude-code` is running with no guard. `make db-dump` truncates the host backup file before the dump runs, so a failed dump can masquerade as a real (empty) backup. On the security side, `gen-env.sh` never re-tightens an already-loose `.env` holding the Postgres superuser password (the live file is mode 644). Finally, two documentation surfaces lag behind reality: the regenerated-each-boot `ENVIRONMENT.md` omits the database entirely, and `CLAUDE.md` does not capture the nuke/profile footgun among its otherwise-meticulous invariants. None of these are crashes, but several are data-correctness and data-destruction hazards that warrant prompt fixes.

## `make nuke` does NOT destroy the DB data volume when the db sidecar is running (docstring is false)

- **Severity / kind:** high / bug
- **Location:** [Makefile:30](Makefile#L30)
- **Evidence:**

  > nuke target = `$(COMPOSE) down -v` where `COMPOSE := docker compose -f .devcontainer/compose.yaml` (no `--profile db`). `db` service has `profiles: [db]` (compose.yaml:54). Live `docker compose down -v --dry-run` (no profile) with claude-db running shows it removes claude-code + claude-pnpm-store/config/bashhistory/workspace but NOT claude-db and NOT claude-pgdata; it then reports `Network claude_default Resource is still in use`. The SAME dry-run WITH `--profile db` removes `Container claude-db` AND `Volume claude-pgdata`.

- **Why it matters:** The nuke docstring promises 'Remove the container AND all named volumes (destroys data)'. When the opt-in db sidecar is up (the normal state once a user has run `make db-up`), `down -v` without the profile leaves `claude-db` running and CANNOT remove the in-use `claude-pgdata` volume — so DB data silently survives a `make nuke` and the network is left dangling. A user who runs `make nuke` to fully reset (e.g. to wipe a corrupted cluster or apply a rotated password the 'real' way) is misled into thinking the data is gone. This is the dangerous direction of the bug: 'destroy' that doesn't destroy.
- **Recommendation:** Point the destructive/lifecycle targets at the profile-aware compose invocation so they cover the sidecar. Change nuke to `$(COMPOSEDB) down -v` (COMPOSEDB already exists at Makefile:3). Doing so makes `down -v` stop+remove claude-db and remove claude-pgdata, as proven by the `--profile db` dry-run. (Compose lists running profiled containers in `ps` but only ACTS on them for down/stop when the profile is active.)

## ENVIRONMENT.md — the authoritative, regenerated-each-boot inventory — omits the Postgres sidecar entirely

- **Severity / kind:** medium / docs
- **Location:** [.devcontainer/seed-claude.sh:46](.devcontainer/seed-claude.sh#L46)
- **Evidence:**

  > The heredoc has sections Host / Languages / Claude Code / Tooling / Persistent volumes, but no Database section; the volume list (lines 82-85) lists 4 volumes and omits `claude-pgdata`. No `db:5432`, `DATABASE_URL`, or `psql` appears anywhere in ENVIRONMENT.md.

- **Why it matters:** seed/CLAUDE.md (the static doc) tells the agent to 'consult ENVIRONMENT.md when you need specifics' and that it is the 'always-fresh companion' with the 'current' state. But ENVIRONMENT.md never mentions the DB, the connection vars, or whether the sidecar is up. An agent that trusts the live snapshot over the static file will conclude there is no database — and the live snapshot is exactly where the *dynamic* fact 'is db reachable right now' belongs (it could probe `pg_isready`/`$DATABASE_URL`). The static seed/CLAUDE.md §9 promises a DB the live inventory contradicts by silence.
- **Recommendation:** Add a Database section to the ENVIRONMENT.md heredoc that prints `$DATABASE_URL` presence, `PGHOST`, and a live `pg_isready -t1` reachability line, plus `claude-pgdata` in the volume list (noting it is the profiled `db` sidecar's volume).

## gen-env.sh never re-secures a pre-existing world-readable .env (early exit skips chmod)

- **Severity / kind:** medium / security
- **Location:** [.devcontainer/gen-env.sh:15](.devcontainer/gen-env.sh#L15)
- **Evidence:**

  > ```
  > if [ -f "$ENV_FILE" ]; then
  >     echo "[gen-env] $ENV_FILE exists — leaving it untouched"
  >     exit 0
  > fi   ...   chmod 600 "$ENV_FILE"  (only reached on the create path, line 44).
  > ```
  > The .env currently on disk is mode 644: `stat -f '%Lp' .devcontainer/.env` => 644.

- **Why it matters:** The umask 077 + chmod 600 only run when .env is freshly created. If .env already exists with loose perms (the live file is 644 — created before the chmod was added, or rewritten by an editor / `cp`, which use the caller's 022 umask), the guard `exit 0`s without ever tightening it. The file holds the Postgres password (PGPASSWORD/POSTGRES_PASSWORD/DATABASE_URL). POSTGRES_USER=claude is created as a DB *superuser* by the official image, and the db sidecar binds 127.0.0.1:5432 on the host. On a multi-user host (the repo explicitly targets native Linux, where shared hosts are plausible) any local user can read the 644 .env and get superuser DB access — which includes COPY ... PROGRAM (command execution as the postgres container user) and large-object file read/write. The script's idempotency claim ("so the password stays stable across rebuilds") silently trades away the security guarantee for any already-loose file.
- **Recommendation:** Always enforce perms on the existing-file path too: before `exit 0`, run `chmod 600 "$ENV_FILE"` (and optionally verify ownership). i.e. move the chmod above the existence check, or add `chmod 600 "$ENV_FILE"` inside the `if [ -f ... ]` branch. Also chmod the live 644 .env now.

## make db-dump truncates/creates the host backup file even when the in-container dump fails

- **Severity / kind:** medium / bug
- **Location:** [Makefile:83](Makefile#L83)
- **Evidence:**

  > `out="db-backups/all-$$ts.sql"; docker exec claude-code pg_dumpall --clean --if-exists > "$$out" && echo "dumped ..."`

- **Why it matters:** The shell opens/truncates `$out` on the host before pg_dumpall runs (redirection happens first). If the db sidecar is down or PG env is missing, `docker exec ... pg_dumpall` writes nothing (or a partial stream then errors), but the file already exists as an empty/truncated backup. The `&&` only guards the success echo, so the operator sees no 'dumped' message yet is left with a zero-byte `all-<ts>.sql` that looks like a real backup in `ls`. A later restore from it silently destroys data (the dump uses --clean). There is also no guard that claude-code or the db sidecar is running.
- **Recommendation:** Dump to a temp file and atomically rename only on success: `docker exec claude-code pg_dumpall --clean --if-exists > "$$tmp" && mv "$$tmp" "$$out" || { rm -f "$$tmp"; echo 'dump failed (is the db sidecar up? make db-up)'; exit 1; }`. Pre-flight that claude-code and claude-db are running.

## `make stop` and `make down` leave `claude-db` running (profile omission, inconsistent with their docstrings)

- **Severity / kind:** medium / bug
- **Location:** [Makefile:24](Makefile#L24)
- **Evidence:**

  > stop = `$(COMPOSE) stop` (Makefile:24), down = `$(COMPOSE) down` (Makefile:27) — both omit `--profile db`. Live `docker compose -f .devcontainer/compose.yaml stop --dry-run` prints only `Container claude-code Stopping/Stopped`; claude-db (Up 5 days, healthy) is untouched. `make down` likewise removes only claude-code.

- **Why it matters:** `make stop` docstring is 'Stop the container (volumes + data preserved)' and `make down` is 'Remove the container'. With the db sidecar running, neither stops claude-db, so `make stop` followed by `make down` leaves an orphaned claude-db holding port 127.0.0.1:5432 and the network. Less dangerous than nuke (no data loss), but it contradicts the stated behavior and can wedge a subsequent `make up`/recreate or confuse a user who thinks 'the container' is fully stopped.
- **Recommendation:** Use the profile-aware invocation for stop/down as well (`$(COMPOSEDB) stop`, `$(COMPOSEDB) down`), or document explicitly that these target only claude-code and the db sidecar must be managed via `make db-down`. Pick one and make stop/down/nuke consistent.

## CLAUDE.md does not document that `make nuke`/`stop`/`down` skip the profiled db sidecar

- **Severity / kind:** low / docs
- **Location:** [CLAUDE.md:137](CLAUDE.md#L137)
- **Evidence:**

  > CLAUDE.md:137-141 documents 'DB password applies only on first init of claude-pgdata ... changing .env later does NOT re-key the running DB. Use `make db-reset` (destroys data)'. `grep -niE 'nuke|down -v' CLAUDE.md` returns nothing — the nuke/profile-omission footgun (Makefile:30) is undocumented, even though the file otherwise meticulously documents load-bearing gotchas (PGDATA subdir, PGHOST empty, pg18 client/server match).

- **Why it matters:** The invariants list is the project's spec for load-bearing behavior. A `make nuke` that fails to destroy DB data when the sidecar is up is exactly the kind of non-obvious, data-correctness gotcha the rest of CLAUDE.md captures. Its absence means a future edit could 'simplify' the Makefile (or trust nuke) without realizing the profile coupling.
- **Recommendation:** If the Makefile bug is fixed (targets switched to `$(COMPOSEDB)`), add a one-line invariant noting the lifecycle targets must use the db-profile compose invocation so they cover the sidecar. If left as-is, document that nuke/stop/down only act on claude-code and the db must be torn down via `make db-down`/`make db-reset`.

## db-* helper targets fail when claude-code is stopped (they `docker exec claude-code` with no guard)

- **Severity / kind:** low / bug
- **Location:** [Makefile:67](Makefile#L67)
- **Evidence:**

  > db-psql (Makefile:67), db-create (Makefile:74,76), db-dump (Makefile:83) all run the pg client via `docker exec claude-code ...`. The pg client lives only in the claude-code image (Dockerfile:69 `postgresql-client-18`), not the db image. Live `docker ps` shows `claude-code  Exited (143)` while `claude-db  Up 5 days (healthy)` — i.e. the db can be up with the client container down.

- **Why it matters:** Because db-up only starts `db` (Makefile:60 `$(COMPOSEDB) up -d db`) and these helpers exec into claude-code, running e.g. `make db-psql` or `make db-dump` after `make stop` (or before the client container is ever started) errors with `Error: No such container: claude-code` / `is not running`. The DB is reachable, but the chosen client host isn't. Minor, but it makes the db helpers depend on an unstated claude-code-running precondition.
- **Recommendation:** Either add a `claude-code` running guard to the db-* targets (e.g. `docker compose ... up -d claude-code` or a `docker inspect` check with a clear error), or run the client inside the db container over the local socket / TCP for the ad-hoc targets. At minimum note the precondition in the docstrings.
